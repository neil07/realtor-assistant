# Reel Agent 原则体系交付报告

> 版本：v1.0 | 截止日期：2026-04-01 | 覆盖范围：Batch 1–5（Phase 1 全部 + Phase 2 第一批）

## 总览

14 条原则，分 5 批次交付。整体从 **均分 5.4** 提升到 **均分 7.1**，核心生存指标（可靠性、成本、验证）全部达到 7 分以上安全线。

| 维度              | 原始均分 | 当前均分 | 提升     |
| ----------------- | -------- | -------- | -------- |
| 产品原则（P1–P6） | 5.2      | 7.0      | +1.8     |
| 架构原则（A1–A8） | 5.6      | 7.1      | +1.5     |
| **总计**          | **5.4**  | **7.1**  | **+1.7** |

## 逐原则对照表

### P1 — 可靠交付

| 项目           | 内容                                                                                                                                                                   |
| -------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **原始评分**   | 6 分                                                                                                                                                                   |
| **当前评分**   | 9 分（+3）                                                                                                                                                             |
| **原来的问题** | 一张照片的 AI 视频渲染失败，整个视频就废了。回调通知发一次没成功就丢了，用户不知道视频做好了。重试没有成本上限，极端情况烧钱不止。                                     |
| **现在的体验** | 单个镜头失败会自动降级为 Ken Burns 运镜（幻灯片式），不影响其他镜头。回调失败会存到本地队列定时重发，不再丢消息。每个 job 设了 $2 成本上限，超了就停止重试、降级交付。 |
| **关键手段**   | 1. per-scene fallback（IMA 超时 → Ken Burns）<br>2. callback 本地重试队列（SQLite 存储 + 定时重发）<br>3. per-job 成本上限（$2 自动停止）                              |

---

### P2 — 越用越懂你

| 项目           | 内容                                                                                                                                                             |
| -------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **原始评分**   | 7 分                                                                                                                                                             |
| **当前评分**   | 8 分（+1）                                                                                                                                                       |
| **原来的问题** | Profile 里记了用户偏好（喜欢什么风格、讨厌什么音乐），但生成视频时完全没读取这些数据。用户满意没修改 = 没有任何反馈信号，系统不知道什么是"好"。                  |
| **现在的体验** | 场景规划和剧本生成时，会读取用户历史偏好（确认过的风格、要避免的音乐、总是要包含的元素），作为创意参考。用户收到视频没要求修改，系统自动把这次的风格记为"好评"。 |
| **关键手段**   | 1. learned_patterns 注入到 plan_scenes + generate_script 的 prompt<br>2. positive feedback 路径（非重试交付 = 正向信号，写入 profile）                           |

### P3 — 主动帮你想着

| 项目           | 内容                                                                                                                                                                                  |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **原始评分**   | 4 分                                                                                                                                                                                  |
| **当前评分**   | 7 分（+3）                                                                                                                                                                            |
| **原来的问题** | 视频生成卡住了，用户完全不知道——只能干等或反复问"好了没"。视频做完就发，没有任何使用建议。生成过程中也没有进度提示，体验像黑箱。                                                      |
| **现在的体验** | 后台 watchdog 每 60 秒检查一次，任何 job 卡住超过 10 分钟自动通知用户。生成过程中关键节点（剧本完成、开始渲染、开始组装）推送进度提示。视频交付时附带社媒文案（caption + hashtags）。 |
| **关键手段**   | 1. watchdog 后台任务（10 分钟卡住 → 告警）<br>2. 关键步骤进度推送（Steps 3/4/5）<br>3. 交付附带 caption                                                                               |

### P4 — 有判断力的协作者

| 项目           | 内容                                                                                                                                                                        |
| -------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **原始评分**   | 4 分                                                                                                                                                                        |
| **当前评分**   | 7 分（+3）                                                                                                                                                                  |
| **原来的问题** | 照片太暗、太糊、全是同一个角度，系统也照做不误——做出来质量很差。经纪人没有任何机会看剧本，只能被动等成品。选风格全靠手动，没有智能推荐。                                    |
| **现在的体验** | 照片分析阶段检测到低质或重复照片，会通过回调建议经纪人补充。脚本生成后有预览暂停点（超时自动继续）。根据物业类型自动推荐风格（luxury → elegant，suburban → professional）。 |
| **关键手段**   | 1. 低质照片检测 + 建议回调<br>2. 脚本预览暂停点<br>3. 物业类型 → 风格自动推荐                                                                                               |

| 项目           | 内容                                                                                               |

| **当前评分**   | 7 分（+1）                                                                                         |

### A4 — 安全验证

| 项目           | 内容                                                                                                                                              |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| **原始评分**   | 5 分                                                                                                                                              |
| **当前评分**   | 8 分（+3）                                                                                                                                        |
| **原来的问题** | API 认证是可选的（没配 token 就跳过），任何人都能调用。视频质量审核只是"看看"，不会阻止低质视频发出去。                                           |
| **现在的体验** | Bearer Token 改为必须，未配置或不匹配直接拒绝请求。视频审核评分低于阈值会阻止交付，推送用户选择（重试/降级/人工接管）。认证事件有结构化审计日志。 |
| **关键手段**   | 1. Bearer Token 强制认证<br>2. Review 评分阈值拦截<br>3. 认证事件结构化日志                                                                       |

| 项目           | 内容                                                                         |
| -------------- | ---------------------------------------------------------------------------- |
| **原始评分**   | 7 分                                                                         |
| **当前评分**   | 8 分（+1）                                                                   |
| **原来的问题** | 有 review_video 质量审核，但只输出诊断报告，不拦截。                         |

### A6 — 成本意识

| 项目           | 内容                                                                                                                |
| -------------- | ------------------------------------------------------------------------------------------------------------------- |
| **原始评分**   | 5 分                                                                                                                |
| **当前评分**   | 7 分（+2）                                                                                                          |
| **原来的问题** | 不知道每个视频花了多少钱。重试没上限，理论上能无限烧钱。相同照片每次都重新上传 IMA，浪费带宽和 quota。              |
| **现在的体验** | 每次 API 调用记录 model + tokens + cost 到 job_logger。per-job $2 成本上限。IMA 上传有 md5 缓存，相同文件不重复传。 |
| **关键手段**   | 1. API cost 记录到 job_logger<br>2. per-job $2 成本上限<br>3. IMA 上传 md5 缓存                                     |

### A8 — 可观测性

| 项目           | 内容                                                                                                                                                                                                                     |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **原始评分**   | 3 分                                                                                                                                                                                                                     |
| **当前评分**   | 6 分（+3）                                                                                                                                                                                                               |
| **原来的问题** | 只有 print 级别的日志，没有结构化记录。`/api/status` 返回的信息很基础，看不出 job 卡在哪、跑了多久。出问题只能翻原始日志文件。                                                                                           |
| **现在的体验** | 关键路径（认证、job 创建、意图分类、bridge 异常）有结构化 JSON 审计日志，输出到独立 `reel_agent.audit` logger。`/api/status` 增加了 elapsed_seconds、step_elapsed_seconds、params_summary，终态 job 不再显示 step 计时。 |
| **关键手段**   | 1. `_structured_log()` 审计日志（JSON → `reel_agent.audit`）<br>2. `/api/status` 增强（耗时、步骤耗时、参数摘要）<br>3. 终态检测（DELIVERED/FAILED/CANCELLED 停止计时）                                                  |
