# Principle - 面向 Agent 的产品设计 5 原则

更新时间：2026-03-26

来源：Anthropic 公开资料 + Claude Code / Agent SDK / MCP 相关文档 + 最近围绕 agent design 的讨论整理

## 一句话结论

从面向人做产品，到面向 Agent 做产品，最大的变化不是界面变了，而是：
**设计对象从“会看页面的人”，变成了“会调用工具、消耗上下文、循环验证、会试错的执行系统”。**

所以，Agent 产品设计的重点，要从 **UI 体验** 转向 **协议、上下文、工具、验证、闭环**。

## 最重要的 5 条原则

### 1. 协议优先，不要只做界面友好

人看页面，Agent 吃协议。

面向人的产品，重点常常是：

- 好不好看
- 好不好点
- 学习成本高不高

面向 Agent 的产品，重点变成：

- 输入输出是否结构化
- 接口是否稳定
- 状态是否可查询
- 错误是否可恢复
- 动作是否可重复调用而不出事故

#### 设计启发

- 核心能力优先做成稳定 action，而不是只能让人点的页面
- 不要让 Agent 靠猜按钮文案、猜页面结构、猜异常含义
- 错误必须 machine-readable，而不是只给一段自然语言

#### 可落地点

- 所有关键动作定义明确 schema
- 返回结构化字段，而不是大段说明文
- 提供错误码、状态码、重试建议
- 高风险写操作默认幂等
- 长任务必须支持状态查询

#### 对比例子

##### 面向人的产品设计

做一个招聘后台的“推荐候选人”页面：

- 用户打开页面
- 点击“搜索”
- 看到候选人列表
- 手动点“保存”“发消息”

重点是：页面清不清楚、按钮好不好找。

##### 面向 Agent 的产品设计

把同样能力设计成可调用协议：

- `search_candidates(jd, filters)`
- `save_to_project(candidate_ids, project_id)`
- `draft_outreach(candidate_id, tone)`

返回：

- `candidates[]`
- `confidence`
- `next_cursor`
- `error_code`

重点是：Agent 能不能稳定调用、重试、继续下一步。

---

### 2. 闭环优先，不要只做单点功能

Agent 不是一次问答，而是一条工作回路：
**Gather context → Take action → Verify → Repeat / Stop**

面向人的产品，经常只要完成一个页面动作就够了。
面向 Agent 的产品，必须支持完整任务闭环。

#### 一个完整 Agent 任务至少要有

- 目标输入
- 上下文读取
- 动作执行
- 结果验证
- 失败恢复
- 中途暂停 / 人工确认
- 最终交付

#### 设计启发

以后不要只问：

- 这个页面要放哪些按钮？

而要问：

- 这个任务是怎么启动的？
- 执行中怎么判断还在推进？
- 失败后怎么恢复？
- 哪些节点必须 ask human？
- 最终交付物是什么？

#### 可落地点

- 每个 Agent 能力都先画任务状态机
- 为长任务设计 checkpoint 和 stop condition
- 输出不是“我做了”，而是“结果、证据、风险、下一步”

#### 对比例子

##### 面向人的产品设计

做一个“职位发布页”：

- 填职位信息
- 点发布
- 完成

这是一个单点功能，只关心当前动作能不能完成。

##### 面向 Agent 的产品设计

设计一条“从 JD 到 shortlist”的完整任务回路：

- 读取 JD
- 搜索候选人
- 初筛
- 生成 shortlist
- 标记风险点
- 人工确认
- 继续外联

重点不是一个页面，而是一条能持续推进的任务链。

---

### 3. 上下文优先，不要让 Agent 靠猜

Agent 的强弱，不只取决于模型本身，更取决于它拿到的上下文是否完整、分层、可调用。

不是“信息越多越好”，而是：
**少噪音，不少关键上下文。**

#### 建议把上下文分成 4 层

##### A. 永久规则层

始终有效的内容：

- 产品目标
- 业务规则
- 禁止事项
- 成功标准

##### B. Skill 层

某类任务的可复用知识与 workflow：

- 怎么做
- 常见坑
- 何时 ask human
- 参考资料

##### C. 任务层

本次任务的特有信息：

- 当前目标
- 输入数据
- 限制条件
- 优先级
- 截止时间

##### D. 环境层

Agent 在这次执行里能访问什么：

- 文件
- 浏览器
- CRM
- 飞书
- 数据库
- 本地命令
- 权限边界

#### 设计启发

Skill + Markdown 这种形式，本质不是“写提示词”，而是在做一层适合 Agent 读取和复用的上下文组织系统。

#### 可落地点

- 长期规则与任务信息分开存
- 高频任务沉淀成 Skill，而不是每次重写 prompt
- 给 Agent 明确暴露工具边界、权限边界、环境边界
- 把成功标准写清楚，别让 Agent 自己脑补

#### 对比例子

##### 面向人的产品设计

给猎头一个系统时，你可以默认他知道：

- 什么叫高匹配候选人
- 什么叫不能碰的公司
- 外联语气该怎么拿捏
- 哪种情况要自己判断

很多东西可以放在培训、经验和默认常识里。

##### 面向 Agent 的产品设计

这些都不能默认，要显式给：

- 长期规则：哪些公司不能挖
- Skill：如何判断匹配度
- 当前任务：本次 JD、地域、年限、排除条件
- 环境：能不能发飞书、能不能查 CRM、能不能打开浏览器

重点不是“多给信息”，而是“把关键信息分层给清楚”。

---

### 4. 工具优先，不要只做功能页

对 Agent 来说，产品能力首先要被设计成“可调用工具”，其次才是“给人看的页面”。

面向人的产品：

- 功能 = 页面 + 按钮 + 表单

面向 Agent 的产品：

- 功能 = 工具能力 + 参数设计 + 返回结构 + 错误恢复

#### 好工具的标准

- 单一职责
- 参数少但完整
- 返回稳定
- 副作用可控
- 可验证
- 可重试

#### 设计启发

不要只问“用户点哪里”，还要问：

- 如果这个能力暴露成 tool，它的名字是什么？
- 它的输入参数有哪些？
- 返回字段够不够 Agent 继续下一步？
- 失败后能不能自动恢复？

#### 可落地点

- 先把核心业务动作拆成 tools / actions
- 每个 tool 都有输入、输出、错误、幂等、审计定义
- 把复杂页面操作，尽可能下沉成稳定 API 或 action

#### 对比例子

##### 面向人的产品设计

给猎头做一个候选人详情页：

- 看简历
- 看推荐理由
- 点“加入项目”
- 点“生成消息”

页面是给人操作的，功能散在多个按钮里。

##### 面向 Agent 的产品设计

把这些页面能力拆成工具：

- `get_candidate_profile(candidate_id)`
- `score_match(candidate_id, jd_id)`
- `add_candidate_to_project(candidate_id, project_id)`
- `generate_outreach_message(candidate_id, style)`

重点是：功能先变成能力，再决定人怎么查看。

---

### 5. 验证优先，不要迷信“一次做对”

面向人时，很多错误由人脑兜底。
面向 Agent 时，如果没有验证机制，系统会悄悄把错误放大。

真正可靠的 Agent 产品，不是不会犯错，而是：
**犯错后能被发现、能被中断、能被纠正、能被复盘。**

#### 高风险动作至少补 5 件事

1. 成功判据
2. 执行前检查
3. 执行后验证
4. 人工 checkpoint
5. 审计日志

#### 设计启发

不要只设计“执行能力”，一定要设计“反馈回路”。

#### 可落地点

- 先定义 success criteria，再让 Agent 动手
- 外发消息、扣费、删除、批量修改等动作默认设人工确认点
- 每个动作记录：看到了什么、做了什么、返回了什么、为什么结束
- 把用户反馈写回记忆，让下一轮更准

#### 对比例子

##### 面向人的产品设计

猎头自己发外联消息时：

- 自己判断要不要发
- 发错了自己发现
- 用经验修正下一封

系统不一定需要很强的校验，因为人脑在兜底。

##### 面向 Agent 的产品设计

Agent 发外联消息前必须过验证：

- 先检查候选人是否已联系过
- 检查联系方式是否有效
- 生成消息草稿
- 在发送前让人确认
- 发送后记录结果：已发 / 失败 / 被拒 / 待跟进

重点不是阻止一切错误，而是让错误可发现、可打断、可修正。

## 三条补充原则

### 6. 人机边界清晰

Agent 负责高频、重复、可验证的部分；
人负责最终判断、关系维护、高风险拍板、价值冲突决策。

原则：
**让 Agent 放大产能，不替代责任。**

### 7. 默认可降级

Agent 系统一定会遇到：权限失效、页面变化、数据缺失、模型误判、工具失败。

所以关键流程都要有降级路径：

- API 失败 → 回退到浏览器
- 自动判断不稳 → 进入人工 review queue
- 信息不足 → ask human
- 批量执行失败 → 先输出草稿而不是硬发

### 8. 先做高频高价值节点，不要上来追求全栈自治

最容易犯的错不是能力不够多，而是没有先收口最小验证路径。

先问：

- 哪个节点最重复？
- 哪个节点最耗时？
- 哪个节点最适合被 Agent 接管？

先做穿一个节点，再扩成链路。

## 这套原则对产品设计的直接指导

以后设计一个 Agent 产品，可以按这个顺序检查：

1. **协议层**：核心动作有没有稳定 schema？
2. **上下文层**：规则、技能、任务、环境有没有分层？
3. **工具层**：能力是不是以 tool/action 的形式存在？
4. **闭环层**：是不是覆盖了启动、执行、验证、恢复、交付？
5. **风控层**：有没有 checkpoint、幂等、日志、降级？

如果这 5 层没补齐，Agent 看起来再聪明，也容易在真实场景里翻车。

## 对招聘产品的直接启发

以 AI Native 猎头工具为例，第一版不应该先做“一个更好看的招聘 SaaS 页面”，而应该优先把一个完整任务做穿：

### 建议优先打穿的任务

**根据 JD 产出可联系候选人 shortlist**

### 可拆成的工具

- `parse_jd`
- `search_candidates`
- `enrich_candidate_profile`
- `score_match`
- `generate_shortlist`
- `draft_outreach_message`
- `ask_human_for_tradeoff`

### 第一版必须有的机制

- 候选人匹配证据
- 匹配理由
- 风险标记
- 人工通过 / 打回
- 打回反馈回写系统
- 下一轮自动修正

这比先堆更多页面，更接近真正的 Agent-native 产品。

## 核心判断标准

以后判断一个产品是不是 Agent-native，不要先看它有没有聊天框，而要看它是否具备：

- 稳定协议
- 分层上下文
- 可调用工具
- 验证闭环
- 人机协作边界

一句话：
**Agent 产品的壁垒，不再只是 UI 和功能，而是协议、上下文、工具、验证、闭环。**
